33/*!
34 \ingroup Util
35 The Process class allows interaction with external processes, such as launching a system command or running an application.
36
37 It can be used as a basic process launcher/monitor, and/or as an input/output device to communicate with a started process via
38 `stdout`, `stderr` and `stdin` streams. It can be used both syncronously and asyncronously. In the latter mode it communicates
39 events back to the user via callback functions (event handlers).
40
41 Instead of typical JavaScript events (like `onload()`), `Process` provides what are called "signals."
42 Which are the same thing with a different name and slightly different syntax.
43
44 In JavaScript signals can be connected-to in the form of `signal.connect(myFunction)` where `signal`
45 is the signal/event function name and `myFunction` is any valid JS function signature like in a callback.
46 Again, like with any 'onevent()' handler from a typical JavaScript object. <br/>
47 For example, to handle the `errorOccurred()` signal/event:
48 ```js
49 const process = new Process();
50 process.errorOccurred.connect((error) => { console.error("Process exited with code:", error); });
51 ```
52
53 \note The Process class is a proxy for [QProcess](https://doc.qt.io/qt-6/qprocess.html),
54 part of the underlying Qt C++ library. Most of the documentation text below is originally from
55 the QProcess documetation (used under the GNU Free Documentation License version 1.3).
56
57 See the [QProcess documentation](https://doc.qt.io/qt-6/qprocess.html) for more details about how `Process` works in general.
58 Some of the enumeration values are documented there (links are provided below in "see also" as needed).
59
60*/

71 /*!
72 \fn int execute(String program, Array<String> arguments = [])
73 \memberof Process
74 Starts the program `program` with the arguments `arguments` in a new process, waits for it to finish, and then returns the exit code of the process.
75
76 Any data the new process writes to the console is forwarded to the calling process. The environment and working directory are inherited from the calling process.
77
78 Argument handling is identical to the respective `start()` overload.
79
80 If the process cannot be started, `-2` is returned. If the process crashes, `-1` is returned. Otherwise, the process' exit code is returned.
81 */
82 Q_INVOKABLE static int execute(const QString &program, const QStringList &arguments = QStringList())
83 {
84 return QProcess::execute(program, arguments);
85 }

87 /*!
88 \fn int startDetached(String program, Array<String> arguments = [], String workingDirectory = "")
89 \memberof Process
90 Starts the program `program` with the arguments `arguments` in a new process, and detaches from it.
91
92 Returns the new process ID (pid) on success; otherwise returns zero. If the calling process exits, the detached process will continue to run unaffected.
93 Argument handling is identical to the respective `start()` overload.
94
95 The process will be started in the directory `workingDirectory`. If `workingDirectory` is empty, the working directory is inherited from the calling process.
96 */
97 Q_INVOKABLE static quint32 startDetached(const QString &program, const QStringList &arguments = {}, const QString &workingDirectory = QString())
98 {
99 qint64 pid = 0;
100 if (QProcess::startDetached(program, arguments, workingDirectory, &pid))
101 return quint32(pid);
102 return 0;
103 }

139 /*!
140 Redirects the process' standard input to the file indicated by `fileName`. When an input redirection is in place, the `Process` object will be in read-only mode (calling `write()` will result in error).
141
142 To make the process read EOF right away, pass `nullDevice()` here. This is cleaner than using `closeWriteChannel()` before writing any data, because it can be set up prior to starting the process.
143
144 If the file `fileName` does not exist at the moment `start()` is called or is not readable, starting the process will fail. Calling `setStandardInputFile()` after the process has started has no effect.
145 */
146 void setStandardInputFile(const QString &fileName) { QProcess::setStandardInputFile(fileName); }
147 /*!
148 Redirects the process' standard output to the file `fileName`. When the redirection is in place, the standard output read channel is closed: reading from it using `read()` will always fail, as will `readAllStandardOutput()`.
149
150 To discard all standard output from the process, pass `nullDevice()` here. This is more efficient than simply never reading the standard output, as no `Process` buffers are filled.
151
152 If the file `fileName` doesn't exist at the moment `start()` is called, it will be created. If it cannot be created, the starting will fail.
153
154 If the file exists and mode is `FS.O_TRUNC`, the file will be truncated. Otherwise (if mode is `FS.O_APPEND`), the file will be appended to.
155
156 Calling `setStandardOutputFile()` after the process has started has no effect.
157 */
158 void setStandardOutputFile(const QString &fileName, FS::OpenMode mode = FS::O_TRUNC) { QProcess::setStandardOutputFile(fileName, toQfileFlags(mode)); }
159 /*!
160 Redirects the process' standard error to the file `fileName`.
161 When the redirection is in place, the standard error read channel is closed: reading from it using `read()` will always fail, as will `readAllStandardError()`.
162
163 To discard all standard error from the process, pass `nullDevice()` here. This is more efficient than simply never reading the standard error, as no `Process` buffers are filled.
164
165 If the file `fileName` doesn't exist at the moment `start()` is called, it will be created. If it cannot be created, the starting will fail.
166
167 If the file exists and mode is `FS.O_TRUNC`, the file will be truncated. Otherwise (if mode is `FS.O_APPEND`), the file will be appended to.
168
169 Calling `setStandardErrorFile()` after the process has started has no effect.
170 */
171 void setStandardErrorFile(const QString &fileName, FS::OpenMode mode = FS::O_TRUNC) { QProcess::setStandardErrorFile(fileName, toQfileFlags(mode)); }

187 /*!
188 \fn void startCommand(String command)
189 \memberof Process
190 \brief Starts the command command in a new process.
191
192 `command` is a single string of text containing both the program name and its arguments. The arguments are separated by one or more spaces. For example:
193 ```js
194 var process = new Process();
195 process.startCommand("del /s *.txt");
196 // same as process.start("del", ["/s", "*.txt"]);
197 ```
198 Arguments containing spaces must be quoted to be correctly supplied to the new process. For example:
199 ```js
200 process.startCommand("dir \"My Documents\"");
201 ```
202 Literal quotes in the command string are represented by triple quotes. For example:
203 ```
204 process.startCommand("dir \"Epic 12\"\"\" Singles\"");
205 ```
206
207 After the command string has been split and unquoted, this function behaves like `start()`.
208 */
209 Q_INVOKABLE void startCommand(const QString &command)
210 {
211 QProcess::startCommand(command);
212 }

238 //! The process may not exit as a result of calling this function (it is given the chance to prompt the user for any unsaved files, etc).
239 //! On Windows, `terminate()` posts a `WM_CLOSE` message to all top-level windows of the process and then to the main thread of the process itself. On Unix and macOS the `SIGTERM` signal is sent.
